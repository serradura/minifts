* **Correction**: Diagramming the lifecycle exposed an imprecision in [search-engine](/architecture/search-engine.md): auto-vacuum was described as firing when a single threshold is crossed, but `vacuum_conditions_met?` requires **both** `dirt_count >= 20` *and* `dirt_factor >= 0.1`. The absolute-count floor is what keeps a tiny index from vacuuming on its first discard, where the ratio alone would already exceed 0.1. Corrected in the prose and the diagram, with the constant cited.

* **Rename**: Renamed the gem from `minisearch`/`Minisearch` to `minifts`/`MiniFTS` (Mini Full-Text Search). RubyGems' name-similarity policy rejects `minisearch` as too close to the pre-existing [`mini_search`](https://rubygems.org/gems/mini_search) gem (both normalize to the same name), so the package could not be published under the old name. The rename covers the gemspec (`minifts.gemspec`, `spec.name`, homepage `serradura/minifts`), the load path (`lib/minifts.rb`, `lib/minifts/`), the `MiniFTS` class and its `MiniFTS::Error`/`WILDCARD`/`VERSION`/`SearchableMap` constants, the `[minifts]` logger prefix and `MiniFTS:` error messages, the full test/benchmark/fidelity suites, and this bundle (files, titles, cross-links). Upstream references are deliberately **unchanged**: the JavaScript library is still `MiniSearch` ([lucaong/minisearch](https://github.com/lucaong/minisearch)), the interop target is still the `minisearch@7.2.0` npm package, and the reference source still lives at `tmp/minisearch/src/`. Public API, scores, and the byte-identical JSON index format are untouched — this is a name change only, `1.0.0` still passes 169 runs / 150,409 assertions.
